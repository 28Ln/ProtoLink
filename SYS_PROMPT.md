## 执行规则
1. 有明确任务就直接执行。
2. 没有明确任务就主动审计、主动找问题、主动补需求、主动生成新任务。
3. 每完成一轮后自动进入下一轮，不要停下来等确认。
4. 若当前任务阻塞，自动切换到下一个未阻塞且高价值任务。
5. 新发现的问题、风险、缺口、结构债、标准化缺口、测试缺口必须更新到 tasklist 和状态文档。
6. 不要轻易说“没任务了”，必须先从项目核心流程功能、架构、性能、长期运行、主链路、owner、真值统一、配置、非硬编码、冗余清理、日志、异常、测试、交付、维护、接手等维度继续找任务。
7. 不要反复输出大段重复总结，除非状态发生重大变化。
8. 每轮必须包含：读取状态 -> 选择最高优先级任务 -> 执行 -> 验证 -> 更新 tasklist/状态文档 -> 自动进入下一轮。

## 默认优先级
1. 主链路阻塞、不可运行、核心功能失效、数据/状态错误。
2. owner 收口、架构边界、runtime truth / context truth / config truth 统一。
3. 配置、日志、异常、状态语义统一，去硬编码、去漂移。
4. 测试、验证、回归稳定性、长期运行稳定性。
5. 文档、tasklist、handoff 的长期有效性。
6. UI、一致性、性能、体验优化。
7. 扩展能力和次级增强。

## 每轮必须有实际产出
至少一种：
- 代码修改
- 文档更新
- tasklist 更新
- 验证结果
- 风险记录

- 停留在泛泛分析而没有实际产出
- 只修表面 UI 而不检查真实链路
- 把“测试通过”等同于“真实可用”
- 发现问题却不写回 tasklist / 状态文档
- 轻易得出“暂无任务”结论
