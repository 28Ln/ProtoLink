# ProtoLink Project Brief

## 1. 项目真实目标

ProtoLink 不是单一串口工具，也不是把参考项目简单拼接。

ProtoLink 的真实目标是：

> 做一个面向 Windows 本地场景的工业通讯与协议调试平台，把串口、Modbus、MQTT、TCP/UDP、自动化规则、脚本联动、数据捕获、配置沉淀统一到一个长期可维护的产品里。

## 2. 为什么不是继续直接改参考项目

`llcom` 和 `Wu.CommTool` 都有强价值，但都不适合作为 ProtoLink 的直接交付底座。

- `llcom` 强在灵活和脚本驱动，但全局状态较重，架构边界弱
- `Wu.CommTool` 强在模块化和工业协议流程，但自动化能力和统一抽象不足
- 两者都偏“功能集合”，ProtoLink 需要的是“平台级结构”

因此，ProtoLink 采用：

- 参考能力吸收
- 架构重新建模
- 根目录独立正式工程

## 3. 产品定位

### 核心定位

- Windows-first 本地桌面应用
- 工业通讯调试与协议工作台
- 强配置、强日志、强自动化、强可扩展

### 目标用户

- 设备联调工程师
- 工控协议开发者
- 自动化测试工程师
- 现场实施与售后支持人员

## 4. 隐含要求补全

基于默会知识，ProtoLink 必须默认满足以下未显式提出但落地必需的要求：

- 标准化：统一传输层、统一消息模型、统一配置格式、统一导入导出策略
- 架构合理性：UI、传输、协议、自动化、存储分层
- UI 一致性：统一导航、统一状态呈现、统一操作入口、统一错误反馈
- 测试完备性：至少覆盖核心模型、配置、工作区、解析器、传输生命周期
- 可维护性：模块边界清晰，避免全局状态蔓延
- 可交付性：必须有运行命令、验证路径、任务台账、里程碑和风险台账
- 安全性：写操作、自动应答、规则联动都要可视、可停、可确认
- 稳定性：长时间运行时不能把 UI 刷死，日志不能无限制吃内存

## 5. 非目标

当前阶段不做：

- Web SaaS
- Linux/macOS 多平台优先级
- 云端账号体系
- 移动端控制台

## 6. 成功标准

ProtoLink 的首个可交付版本至少要满足：

- 能稳定启动并初始化工作区
- 串口、TCP、UDP、MQTT 中至少一个传输链路跑通到可视化日志
- 原始字节与解析结果双轨保存
- 配置可保存、可导入、可导出
- 至少一个协议工作流端到端可用，首选 Modbus RTU
